# 工具栈选型(2026 SOTA)

这一篇解释**为什么本仓选这一套**而不是别的。每个决定都对应面试可能被问到的问题。

## 1. SFT/DPO/GRPO:为什么用 Unsloth + Liger Kernel?

### Unsloth

- **核心价值**:单卡显存降 50%+、吞吐提 30%+,在 12GB 卡上能塞下 1.5B 全参 SFT
- **怎么做到的**:custom Triton kernel + 手动 ckpt 优化 + 4bit base + LoRA 友好的内存布局
- **取舍**:多卡支持差(社区版),生产环境换 Axolotl;但单卡教学场景是无脑选

### Liger Kernel(LinkedIn 开源)

- **核心价值**:把 RMSNorm / RoPE / SwiGLU / cross-entropy 这些算子 fuse 进 Triton kernel
- **数字**:HF 官方测,Llama-3 8B SFT,**+20% 吞吐,-60% 显存**(主要是 fused CE 不再实例化 logits)
- **怎么用**:`from liger_kernel.transformers import apply_liger_kernel_to_qwen2; apply_liger_kernel_to_qwen2()`,一行替换

### 为什么不只用 TRL 原生

TRL 原生也能跑,但:
- 显存浪费(尤其 cross-entropy 那一大块 logits 张量)
- 没有 fused RoPE / RMSNorm,慢
- 12GB 卡跑 1.5B 全参 SFT 会 OOM,加了 Liger 就稳

### 工具对比(社区共识,2026-05)

| 工具 | 单卡省显存 | 多卡 | UI | 适合场景 |
|---|---|---|---|---|
| **Unsloth** | ⭐⭐⭐⭐⭐ | ⭐⭐ | CLI | 单卡研发 / 教学(本仓首选) |
| Axolotl | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | YAML | 多卡生产 |
| LLaMA-Factory | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | WebUI | 不写代码、要点点鼠标的同学 |
| TRL 裸用 | ⭐⭐ | ⭐⭐⭐ | API | 学习内部机制 |

## 2. GRPO 为什么用 verifiable reward 不做偏好对?

### 偏好对(传统 RLHF/DPO 流派)

- 流程:人工标注 chosen vs rejected → 训 reward model → 用 RM 给生成打分 → PPO/DPO 更新
- 问题:**reward model 自己就是个偏差源**(人对中间过程评分不靠谱),训得越久越偏

### Verifiable reward(2025+ 共识,DeepSeek-R1 / nanochat / Open-R1 路线)

- 流程:任务有 ground truth(数学题答案、代码 testcase)→ 规则化 reward(答对 +1)→ GRPO 直接更新
- 优点:reward 永远准确,无偏差累积;在数学/代码/逻辑任务上效果远超偏好对
- 缺点:只适用于可验证任务,聊天/创作类还得偏好对

### 本仓选择

- 06 章 02 节:保留 DPO 偏好对(教学价值 + 通用对齐场景)
- 06 章 03 节 + 09 章 capstone:**主推 verifiable reward GRPO**(GSM8K-zh,答案精确匹配)

## 3. 推理部署四条路怎么选?

| 路径 | 用什么 | 何时用 |
|---|---|---|
| **基线** | `transformers.generate()` | 看 baseline 数字 |
| **服务化** | `vLLM`(PagedAttention) | 中小规模线上 / 研发 |
| **加速** | `speculative decoding`(draft+target) | 同精度下要 1.5-2x |
| **压缩** | `GPTQ / AWQ / GGUF` | 边端 / CPU / 极致显存 |
| **生产** | `TensorRT-LLM` | NVIDIA 大规模线上,本仓只读概念 |

国内面试问得最多的是 vLLM(PagedAttention)和量化(GPTQ vs AWQ),这两个一定要能徒手讲清楚原理。

## 4. 评测怎么用?

- **lm-eval-harness**:基线 / 横向对比的事实标准,backend 选 `vllm` 才快
- **中文**:CMMLU + C-Eval(STEM/人文/社科分项)
- **数学**:GSM8K(`flexible-extract` 模式)+ GSM8K-zh(本仓用)
- **不要只看 perplexity**(SFT 后 PPL 通常会变差,但 chat 能力反而升)

## 5. 数据怎么处理?

- **小规模(< 100MB)**:`datasets` + 手写过滤即可
- **中规模(GB 级)**:`datasketch` MinHash + 简单 quality filter
- **大规模(TB 级,工业)**:`datatrove`(HF 官方)+ Spark / Ray;本仓 02 章只演示 datatrove 思路
